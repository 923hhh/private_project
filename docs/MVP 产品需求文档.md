## 1. 文档目的

本文档在《系统架构文档》之下，给出 **首期 MVP 的软件功能需求**：功能模块划分、**可排期的子需求（FR-xx-yy）**、优先级（必选 / 可选 / 预留）、前置/后置/失败条件、关键验收要点及与用例（UC）的映射。拆分任务与联调时，以 **`http://127.0.0.1:8000/docs` OpenAPI** 与 **`backend/tests/`** 为准。

## 2. 范围与假设

### 2.1 产品范围

- **首期试点**：机泵类设备辅助检修；B/S 交付；先检索后生成。
- **MVP 边界**与《系统架构文档》**§5.8 首版实现边界**一致：本文以 **必选 + 可选** 列需求；**预留**能力仅列条目不展开验收。

### 2.2 假设

- 已具备或可导入设备台账、SOP 片段、案例文本等冷启动知识；赛题参考手册可作为知识来源之一。
- 四类角色（一线工人、检修班长/专家、安全审批人、系统/知识管理员）已在系统中配置或可后台创建。

### 2.3 不在 MVP 内

- PLC 控制、自动启停、全厂设备统一接入、ERP 集成。

### 2.4 首版最小演示闭环（MVP 主线）

**首版至少须能完整演示以下一条链路**（答辩与「MVP 成立」判定以此为主线；其它功能为该主线上的增强或分支）：

**选择设备 → 上传图片与/或文字提问 → 混合检索并返回带出处的建议 → 创建/绑定工单并记录检索快照 →（可选分支）进入工步引导 →（可选分支）高危工步触发审批且未通过不可推进 → 结果回填（§2.6.1 结构化字段 + 至少一张凭证图 + 可选补充说明）→ 专家复核后结单 → 从工单生成候选知识条目并由专家审核、管理员发布 → 发布后检索可命中**

说明：

- 「可选分支」在演示脚本中可择一二展示，但 **主线前半段（设备—提问—检索—出处—工单—快照）与后半段（回填—结单—候选知识—审核发布）须 P0 全通**。
- 审批分支若演示资源不足，须在团队记录或 `backend/tests/` 用例说明中标注为「抽样演示」，但 **接口与后端规则仍须 P0 实现**。

### 2.5 暂不做但已考虑（刻意不纳入首版）

以下能力已在架构或业务中讨论，**首版不验收**，避免评委误认为「遗漏」：

| 方向 | 说明 |
| ---- | ---- |
| 多租户 / SaaS 化 | 单企业、单库部署即可 |
| 全厂多设备类型统一接入 | 首期仅机泵试点 |
| 完整离线同步队列 | 仅架构预留，首版可不做 |
| 自动化 MLOps、批量训练导出流水线 | FR-ANN-04 为 P2 |
| 预测性维护、时序诊断联动 | 与现有仓库示例域分离，不并入本 MVP 验收 |
| 本地大模型热切换集群 | 首版默认云端 API，本地为可选 |

### 2.6 接口与数据形态拍板（首版固定，不二选一）

以下约定 **全文唯一口径**，OpenAPI（`/docs`）与持久化模型须与此一致；若变更须走 §12 需求变更并升版本号。

#### 2.6.1 结果回填：以结构化字段为主

| 约定 | 内容 |
| ---- | ---- |
| 主形态 | **结构化字段必填**；**补充说明**为可选自由文本，避免接口只剩长文本难以校验与统计。 |
| 必填结构化字段 | `resolution_status`：`resolved` \| `unresolved`；`closure_code`（枚举见下）；`attachment_ids`（**至少 1 个**现场凭证附件）。 |
| 可选自由文本 | `detail_notes`：字符串，**可空**；最大长度 **2000 字符**（UTF-8 长度按实现约定在接口文档写死）。 |
| `closure_code`（`resolved` 时） | 必填，取值：`NORMAL` \| `PART_REPLACED` \| `ADJUSTED` \| `OTHER`。选 `OTHER` 时：若 `detail_notes` 为空则 **400**（强制留痕）。 |
| `closure_code`（`unresolved` 时） | **固定为 `UNRESOLVED`**，且 **必须由请求体显式传入** `UNRESOLVED`（禁止省略后由服务端静默默认，避免前后端与抓包口径不一致）。 |
| `unresolved` 时另必填 | **`post_unresolved_action`**，枚举 **仅** §2.6.2 三值；**`unresolved_reason_code`** 必填，枚举见 §2.6.2 表下。 |

#### 2.6.2 未解决时后续动作：三个固定选项（枚举唯一）

字段名建议：`post_unresolved_action`（仅当 `resolution_status=unresolved` 时必填，否则须 **400**）。

| 枚举值 | 含义 | 期望工单侧行为（由 FR-WO 状态机落地，接口只传枚举） |
| ------ | ---- | ------------------------------------------------------ |
| `REOPEN_ESCALATION` | **再次发起升级会诊** | 创建或关联新的升级会话，工单进入「待升级/会诊中」等流程文档规定状态 |
| `RETRY_RETRIEVAL` | **补充信息与材料后重新检索** | 回到允许再次提问/检索的状态（如回到 S3 或专设「待补充重试」状态，以《检修业务流程》修订为准，**接口文档写死映射**） |
| `CLOSE_UNRESOLVED` | **未解决结案** | 进入 `SX` 或流程文档定义的「未解决关闭」终态；`unresolved_reason_code` 仍 **必填**（与 §2.6.1 一致）；**建议** `detail_notes` 非空；若 `unresolved_reason_code=OTHER` 则 **`detail_notes` 必填**，否则 **400** |

`unresolved_reason_code` 首版枚举示例：`EQUIPMENT_LIMIT` 设备条件不足 \| `INFO_INSUFFICIENT` 信息仍不足 \| `EXPERT_REQUIRED` 需线下专家 \| `USER_ABORT` 用户主动中止 \| `OTHER`。

#### 2.6.3 重复审批：幂等成功策略

对同一 `approval_task_id`：

| 情形 | HTTP | 行为 |
| ---- | ---- | ---- |
| 任务已为终态，客户端 **重复提交与当时终态相同的结论与相同关键参数** | **200** | 响应体返回当前审批任务终态；`business_code` = **`ALREADY_PROCESSED`**；**不新增**第二条终态审批记录；**可**追加一条审计「幂等重复提交」（推荐，便于溯源） |
| 任务已为终态，客户端提交 **与已生效结论不一致** 的新结论 | **409** | `business_code` = **`CONFLICT`**，不修改原记录 |
| 任务非终态 | 按正常审批写库 | **201** 或 **200** 由接口文档统一 |

## 6. 详细功能需求

### 6.1 认证与会话（FR-AUTH）— P0

| 子 ID | 描述 |
| ----- | ---- |
| FR-AUTH-01 | 用户可使用账号密码登录；错误次数限制与锁定策略可配置（首版可简化）。 |
| FR-AUTH-02 | 登出后会话失效；敏感操作需有效会话。 |
| FR-AUTH-03 | 密码与密钥不以明文写入日志。 |

**验收要点**：未登录访问业务 API 返回 **401**；登录后获得角色上下文。

---

### 6.2 权限（FR-RBAC）— P0

| 子 ID | 描述 |
| ----- | ---- |
| FR-RBAC-01 | 系统预置四类角色，权限矩阵与本章 UC 描述一致。 |
| FR-RBAC-02 | 接口级校验：无权限返回 **403**；高危工步推进须校验审批状态（与架构 §8.1 一致）。 |

**验收要点**：矩阵中「❌」的操作在 API 层不可成功。

---

### 6.3 设备台账（FR-DEV）— P0

| 子 ID | 描述 |
| ----- | ---- |
| FR-DEV-01 | 维护设备主数据：设备类型、型号、编号、安装位置等（字段以《数据字典》为准）。 |
| FR-DEV-02 | 发起检修/提问时可选择或绑定设备，作为检索过滤条件。 |
| FR-DEV-03 | **责任专家绑定**：每台设备须能配置 **`responsibility_expert_user_id`**（可为空，但为空时 **不得** 满足 FR-ESC-05 发起条件）；与升级会诊分配一致。 |

**验收要点**：检索请求可携带设备上下文，并在审计/快照中可追溯；升级发起前可校验责任专家已配置。

---

### 6.4 附件与上传（FR-ATT）— P0

| 子 ID | 描述 |
| ----- | ---- |
| FR-ATT-01 | 支持上传图片（首版必选）；支持常见格式（如 JPEG/PNG）；单文件 **≤ 10 MB**（与《系统架构文档》§10.1 一致）。 |
| FR-ATT-02 | 上传失败保留工单草稿或明确错误码，可重试（与架构 §7.1 一致）。 |
| FR-ATT-03 | 附件元数据与工单绑定，可列表查看与下载（权限内）。 |

**验收要点**：超限拒绝；成功落盘且 DB 可查 `attachments` 记录。

---

### 6.5 语音转写（FR-ASR）— P1

| 子 ID | 描述 |
| ----- | ---- |
| FR-ASR-01 | 用户上传录音后，系统可异步转写为文本并参与检索（实现可选用云端 ASR 或本地）。 |
| FR-ASR-02 | **未启用 ASR（P1 未做）时**：上传录音入口隐藏或灰显；若用户仅有录音文件，界面 **明确提示改用文字输入**，不得静默失败。 |
| FR-ASR-03 | 转写失败不阻塞工单创建：工单仍可保存，检索仅使用已有人工输入文本。 |

**验收要点**（若做 P1）：转写任务状态可查；失败不阻塞工单创建。

---

### 6.6 检索与 RAG 对话（FR-RAG）— P0

大模块拆分为可独立排期、可独立测试的子需求：

| 子 ID | 描述 |
| ----- | ---- |
| FR-RAG-01 | **混合检索**：语义向量检索 + 关键词/BM25 或等价模糊匹配；支持设备上下文过滤。 |
| FR-RAG-02 | **Top-K 与出处**：返回片段须带 `source_document`、`chunk_id`（或等价字段），供前端与快照落库。 |
| FR-RAG-03 | **空命中与低置信度**：无命中或低于阈值时禁止杜撰长答案；返回短提示并引导升级或补充信息；可驱动工单进入待升级状态。 |
| FR-RAG-04 | **引用与 100% 可追溯**：对外展示的「建议」正文须能关联到已返回片段；无法满足时仅输出短提示，不输出无引用长文。 |
| FR-RAG-05 | **检索快照**：每次对外返回建议时写入 `retrieval_snapshots`，与工单/会话绑定并冻结引用版本。 |
| FR-RAG-06 | **模型失败降级**：模型超时或失败时返回 Top-K 片段列表 + 业务码，HTTP **不以裸 500** 暴露给前端（与架构 §7.1 一致）。 |

**验收要点**：分别构造「仅检索」「空命中」「模型失败」用例；快照表有记录且与工单 ID 关联。

### 6.7 流式输出（FR-SSE）— P1

| 子 ID | 描述 |
| ----- | ---- |
| FR-SSE-01 | 可选：对长文本生成使用 SSE 或等价流式协议，提升首字体验。 |
| FR-SSE-02 | 流式结束时须携带完整引用信息或 `retrieval_snapshot_id`。 |
| FR-SSE-03 | **未实现 SSE（P1 未做）时**：采用 **同步请求 + 前端「处理中」状态**（转圈/文案），或 **轮询任务结果接口**；禁止无反馈长时间白屏；首条建议仍须满足 §7 响应时间口径或测试报告注明环境偏差。 |

### 6.8 工单（FR-WO）— P0

**模块边界**：**FR-WO 仅负责**工单主记录、**状态机合法迁移**、**事件流水**。**不包含**结单业务字段的填写的业务规则细节——后者归 **FR-FILL**。

| 子 ID | 描述 |
| ----- | ---- |
| FR-WO-01 | 工单主记录：创建、查询、列表；与设备、当前状态编码绑定。 |
| FR-WO-02 | 状态迁移与代码中工单状态机（S1～S12、SX）一致；非法迁移 **拒绝** 并返回可读原因。 |
| FR-WO-03 | `work_order_events`：记录每次迁移、操作者、时间、备注（及关联审批单 ID 等外键视实现而定）。 |

**验收要点**：主流转与流程文档 **§6.1** 一致；非法迁移不可落库。

---

### 6.9 作业引导（FR-GUIDE）— P0

| 子 ID | 描述 |
| ----- | ---- |
| FR-GUIDE-01 | **等级与模板**：选择检修等级或默认值；加载 `flow_template_id` 对应工步序列。 |
| FR-GUIDE-02 | **工步展示**：逐步展示操作说明、工具备件、安全合规文案。 |
| FR-GUIDE-03 | **工步完成校验**：标记「本步完成」须服务端校验；高危未审批工步 **拒绝** 推进（与 FR-APP 联动）。 |
| FR-GUIDE-04 | **模板治理**：专家/管理员按权限维护模板草稿；知识管理员发布模板版本（与权限矩阵一致）。 |

**验收要点**：与 UC-06 一致；绕过前端的非法完成请求被 API 拒绝。

### 6.10 高危审批（FR-APP）— P0

| 子 ID | 描述 |
| ----- | ---- |
| FR-APP-01 | 工步或服务端规则判定为高危时，创建 `approval_tasks`，工单进入待审批状态。 |
| FR-APP-02 | 审批人批准/驳回/要求补充材料；每次操作写审计。 |
| FR-APP-03 | 未审批通过前，禁止进入「检修中」或允许执行高危工步的等效状态。 |
| FR-APP-04 | 审批超时：可配置提醒；**默认不自动放行**（与架构 §7.2 一致）。 |

**前置 / 输入 / 后置 / 失败（评审钉）**

| 维度 | 说明 |
| ---- | ---- |
| **前置条件** | 当前工单处于需审批状态；目标工步 `requires_approval=true` 或规则引擎命中高危。 |
| **输入** | 工单 ID、工步序号或工步 ID、审批结论、可选补充说明、可选现场材料附件 ID 列表。 |
| **后置条件** | `approval_tasks` 新增一条终态记录；工单状态按流程迁移；`work_order_events` 追加一条。 |
| **失败条件** | 操作者无审批角色 → **403**；材料不满足「补充材料」要求 → **400** 与明确字段提示；工单状态与工步序号不匹配 → **409** 业务冲突；**重复审批** → 按 **§2.6.3**（同结论 **200 + ALREADY_PROCESSED**；不同结论 **409 CONFLICT**）。 |

### 6.11 升级会诊（FR-ESC）— P0

| 子 ID | 描述 |
| ----- | ---- |
| FR-ESC-01 | 一线发起升级；系统汇总上下文、首轮建议、附件链接、检索快照 ID。 |
| FR-ESC-02 | 专家补充/修订建议；若含高危动作则进入 FR-APP。 |
| FR-ESC-03 | 结论回传一线，并写工单事件；升级会话置为已结论（终态枚举由接口文档定义）。 |
| FR-ESC-04 | **重复发起控制**：同一工单存在**进行中**的升级单时，再次发起返回 **409** `ESCALATION_IN_PROGRESS`（前端可辅助灰显，以服务端为准）。 |
| FR-ESC-05 | 创建升级单时必须写入 **`assigned_expert_id`**：**取自设备台账绑定的「责任专家」用户 ID**；若设备未绑定责任专家则 **400** `EXPERT_NOT_CONFIGURED`，不允许发起升级（须先由管理员维护设备—专家映射）。 |

**前置 / 输入 / 后置 / 失败**

| 维度 | 说明 |
| ---- | ---- |
| **前置条件** | 工单状态允许升级（如 **S4/S5** 或流程文档规定的「待升级/会诊」路径）；或系统判定 **空命中/低置信度** 后由规则允许一线从 S3 等状态一键升级（**具体允许集合与流程文档一致，接口文档列状态白名单**）。 |
| **输入** | `work_order_id`、一线 **补充说明** `escalation_note`（建议必填，最少 10 字，**400** 若不足）、**附件 ID 列表**（可选，与 FR-ATT 关联）、可选 `related_message_id`（指向某条模型回答）。 |
| **后置条件** | 创建 `escalation`（或等价表）记录，状态「待专家处理」；`work_order_events` 追加；通知专家（站内列表即可，站内信/Push 为 P1）。 |
| **失败条件** | 无发起权限 → **403**；工单状态不在允许集合 → **409**；**进行中升级已存在** → **409** `ESCALATION_IN_PROGRESS`（与 FR-ESC-04 一致）；工单不属于当前用户（一线）→ **403**；专家处理时 **非 `assigned_expert_id`** → **403**。 |

**验收要点**：与 UC-03 一致；升级单与工单关联可查；重复发起与非法状态可被自动化用例覆盖。

### 6.12 结果回填与结单（FR-FILL）— P0

**模块边界**：**FR-FILL 负责**在允许的状态下 **采集结单所需业务结果**（描述、是否解决、凭证等）及 **校验**；**不负责**状态机规则定义（归 FR-WO）。

| 子 ID | 描述 |
| ----- | ---- |
| FR-FILL-01 | 在工单处于 **§5.1 与流程文档一致的允许回填状态**（首版为 **S8**）时开放回填表单。 |
| FR-FILL-02 | **结构化必填**：`resolution_status`、`closure_code`（规则见 **§2.6.1**）、`attachment_ids`（≥1）；**可选** `detail_notes`。未满足 → **400** 及字段级错误。 |
| FR-FILL-03 | `resolution_status=unresolved` 时 **必填** `post_unresolved_action`，取值 **仅允许** §2.6.2 三个枚举；须校验 `unresolved_reason_code` / `detail_notes` 与 §2.6.2 的填写规则。 |
| FR-FILL-04 | 专家复核通过后触发 FR-WO 侧合法迁移至结单相关状态。 |

**前置 / 输入 / 后置 / 失败**

| 维度 | 说明 |
| ---- | ---- |
| **前置条件** | 工单状态 = **S8**（若流程变更，与 §5.1 脚注同步）；工步链未闭环是否禁止回填——**首版拍板：不强制校验工步链全部完成，以能否进入 S8 为准**（若业务后续收紧，改流程文档与本条）。 |
| **输入** | `work_order_id`、§2.6.1 所列结构化字段、`attachment_ids`、`post_unresolved_action`（条件必填）、`unresolved_reason_code`（条件必填）。 |
| **后置条件** | 回填记录持久化；`work_order_events` 追加；进入待专家复核（如 S9）。 |
| **失败条件** | 状态非 S8 → **409**；违反 §2.6.1 / §2.6.2 → **400**；权限不足 → **403**。 |

**验收要点**：与 UC-02/05、状态 S8～S10 一致。

### 6.13 知识库（FR-KB）— P0

**与 FR-ANN 的边界**：**FR-ANN** 负责「对模型输出发现问题、打标注、可选推送修订入口」；**FR-KB** 负责「知识治理流水线：候选 → 审核 → 发布/回滚/下架 → 参与检索」。标注产生的修订须 **进入 FR-KB 的候选/审核队列**，不得绕过审核直接写入正式库。

| 子 ID | 描述 |
| ----- | ---- |
| FR-KB-01 | 从已结单工单 **生成候选知识条目**（草稿），带来源工单 ID 与快照引用。 |
| FR-KB-02 | 专家 **审核**（通过/驳回/需修改）；**不**具备最终发布权（与架构权限隔离一致）。 |
| FR-KB-03 | 知识管理员 **发布 / 回滚 / 下架**；发布后参与 FR-RAG 检索。 |
| FR-KB-04 | 知识条目版本号单调递增；旧版本可查不可伪删。 |

## 10. MVP 完成判定标准（Definition of Done）

**首版 MVP 视为完成**，须同时满足：

1. **P0 子需求全部可实现且可演示**（允许已知轻微 UI 缺陷，但须在团队记录中登记）。
2. **§2.4 最小演示闭环**至少完整跑通 **一条** 无断点录屏路径（含设备—提问—检索—出处—工单—快照—回填—结单—候选知识—审核—发布—再检索命中）。
3. **FR-APP** 与 **FR-GUIDE-03** 在服务端对高危工步 **二次校验** 生效（可用 Postman 构造非法请求验证 **409/403**）。
4. **来源可追溯率**：凡有「模型组织类建议」返回，均满足 FR-RAG-04；空命中符合 FR-RAG-03。
5. **FR-RAG-06** 与 **FR-RAG-03** 的降级/空命中行为在验收用例中 **各至少 1 条** 通过。
6. **FR-AUD-01** 覆盖审批、知识发布、标注、导出中的 **至少三类** 操作可追溯查询。

**不纳入 DoD**：FR-ASR、FR-SSE、FR-ANN-04 的完整实现（P1/P2）；但若实现 P1，须在团队记录中注明行为与降级策略已符合本文 §6.5、§6.7。

## 11. 推荐开发顺序（排期用）

便于团队按 MVP 主线递进，**非并行乱做**；同一行内可前后微调，但不宜跳行实现业务闭环。

| 阶段 | 建议顺序 | 说明 |
| ---- | -------- | ---- |
| 1 | FR-HLT → FR-AUTH → FR-RBAC → FR-ADM（最小用户种子） | 先有可部署骨架与身份 |
| 2 | FR-DEV → FR-ATT | 设备上下文与上传为检索前置 |
| 3 | FR-RAG → FR-WO | **核心演示链**：检索 + 快照 + 工单主记录与状态迁移 |
| 4 | FR-FILL → FR-KB | 回填结构化（§2.6）与结单后候选知识 |
| 5 | FR-APP → FR-GUIDE → FR-ESC | 高危阻断、工步模板、升级会诊（依赖工单与检索已稳） |
| 6 | FR-ANN → FR-AUD | 标注与审计查询收尾 |
| 7 | P1 | FR-ASR、FR-SSE 按时间插入，不阻塞 P0 闭环 |
