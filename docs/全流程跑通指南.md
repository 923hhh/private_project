# 前端全流程跑通指南（当前版本）

本文面向**现在这套前端**，目标是把页面从“能打开”跑到“能真实联后端、能登录、能创建任务、能查看详情、能跑工单/案例链路”。

适用环境：

- Windows
- PowerShell

---

## 1. 先说当前版本的两个关键事实

### 1.1 推荐启动方式已经变了

现在最推荐的不是分别手工起前后端，而是直接用：

```powershell
.\scripts\start-dev.ps1
```

这个脚本会自动做几件事：

- 启动后端 `uvicorn`
- 启动前端 `next dev`
- 自动把 `frontend/.env.local` 里的 `NEXT_PUBLIC_API_BASE_URL` 改成当前后端地址

### 1.2 当前一键脚本使用的后端端口是 `18000`

不是旧文档里的 `8000`。

也就是说，如果你走一键脚本，当前默认地址是：

- 前端：`http://127.0.0.1:3000`
- 后端：`http://127.0.0.1:18000`
- 后端文档：`http://127.0.0.1:18000/docs`

如果你手工启动后端到别的端口，必须同步修改：

```text
frontend/.env.local
```

里的：

```text
NEXT_PUBLIC_API_BASE_URL=...
```

## 3. 推荐跑通方式：一键启动

在仓库根目录执行：

```powershell
.\scripts\start-dev.ps1
```

预期结果：

- 弹出两个 PowerShell 窗口
- 一个跑后端
- 一个跑前端

终端最终会提示：

- 后端文档：`http://127.0.0.1:18000/docs`
- 前端地址：`http://127.0.0.1:3000`

### 3.1 一键启动会自动处理的事

- 检查 `frontend/.env.local`
- 自动写入或更新：

```text
NEXT_PUBLIC_API_BASE_URL=http://127.0.0.1:18000
```

- 如果 `frontend/node_modules` 不存在，会自动执行 `npm install`

## 4. 手工启动方式（只在需要单独排查时用）

### 4.1 启动后端

```powershell
cd backend
..\venv\Scripts\python.exe -m uvicorn app.main:app --reload --host 127.0.0.1 --port 18000
```

健康检查：

- 打开 `http://127.0.0.1:18000/health`

应返回类似：

```json
{"status":"...","database":"..."}
```

### 4.2 启动前端

```powershell
cd frontend
npm run dev
```

前端默认访问：

- `http://127.0.0.1:3000`

### 4.3 确认前端环境变量

文件：

```text
frontend/.env.local
```

内容至少应包含：

```text
NEXT_PUBLIC_API_BASE_URL=http://127.0.0.1:18000
```

## 7. 当前前端全流程验收顺序

下面这条路径是最短、最稳的前端验收路线。

### 7.1 总览页 `/dashboard`

访问：

- `http://127.0.0.1:3000/dashboard`

检查点：

- 页面能打开
- 顶栏导航正常
- 总览卡片能加载
- 无明显白屏或接口报错

### 7.2 诊断任务列表 `/tasks`

访问：

- `http://127.0.0.1:3000/tasks`

检查点：

- 任务列表能加载
- 搜索框、筛选、分页可正常显示
- “新建诊断任务”弹窗可打开

关键接口：

- `GET /api/v1/history`
- `POST /api/v1/tasks`
- `DELETE /api/v1/tasks/{id}`

### 7.3 创建一个诊断任务

在 `/tasks` 中创建任务，至少填写：

- 设备类型
- 故障现象 / 观察说明

创建成功后，检查：

- 列表出现新任务
- 任务状态能显示
- 点击能进入详情页

### 7.4 任务详情 `/tasks/{id}`

进入刚创建的任务，例如：

- `http://127.0.0.1:3000/tasks/1`

或处理视图：

- `http://127.0.0.1:3000/tasks/1?action=process`

当前前端详情页已重构，重点检查：

- 页头：任务编号、设备对象、状态、分享/导出/重新运行
- 左侧：任务概览、关键读数、当前建议入口
- 右侧：诊断结论主卡、执行时间线

关键接口：

- `GET /api/v1/tasks/{id}`
- `GET /api/v1/agents/assist/stream?...`（SSE）
- `PATCH /api/v1/tasks/{id}/execution-timeline`
- `GET /api/v1/export/{id}`

关键验收点：

1. 打开详情页后能拉到任务详情
2. 运行中状态下时间线会增长
3. 完成后导出按钮可点
4. 刷新详情页后，执行时间线仍存在

### 7.5 案例页 `/cases`

访问：

- `http://127.0.0.1:3000/cases`

关键接口：

- `GET /api/v1/cases`
- `POST /api/v1/cases`
- `POST /api/v1/knowledge/imports`

检查点：

- 列表能加载
- 更多筛选能打开
- “新建案例”弹窗能提交
- “知识文档上传”抽屉能打开

### 7.6 监控页 `/monitoring`

访问：

- `http://127.0.0.1:3000/monitoring`

关键接口：

- `GET /health`
- `GET /api/v1/system/metrics`

检查点：

- 设备列表能显示
- 系统健康区能展开
- 日志弹窗能打开

### 7.7 工单页 `/tickets`

访问：

- `http://127.0.0.1:3000/tickets`

关键接口：

- `GET /api/v1/maintenance/work-orders`
- `GET /api/v1/maintenance/devices`
- `POST /api/v1/maintenance/work-orders`

检查点：

- 登录后能正常进入工单页
- 列表能同步真实工单
- “新建检修工单”弹窗能拉到设备列表

注意前置条件：

- 必须已经登录
- 后端必须至少存在 1 条设备数据，否则无法创建工单

## 8. 当前版本最容易踩的坑

### 8.1 页面还在走“演示模式”数据

虽然 UI 上的“演示模式/真实模式”开关已经在逐步移除，但代码里仍然保留了 `demo-mode` 逻辑。

前端判断位置在：

```text
frontend/lib/demo-mode.ts
```

以及：

```text
frontend/lib/api.ts
```

如果你之前开过演示模式，本地可能残留：

```text
localStorage["faultdiag.demo_mode"] = "1"
```

会导致部分接口直接走 demo 分支。

### 8.2 如何强制切回真实模式

打开浏览器控制台执行：

```js
localStorage.removeItem("faultdiag.demo_mode")
location.reload()
```

如果你想更稳一点，也可以顺手检查环境变量里**不要**设置：

```text
NEXT_PUBLIC_DEMO_MODE=true
```

### 8.3 工单页空白或 401

优先检查：

1. 是否先登录了 `/login`
2. 本地 token 是否存在
3. 后端检修域接口是否可达
4. `NEXT_PUBLIC_API_BASE_URL` 是否真的指到后端

### 8.4 一键脚本启动成功，但前端仍请求旧端口

检查：

```text
frontend/.env.local
```

确认已经被改成：

```text
NEXT_PUBLIC_API_BASE_URL=http://127.0.0.1:18000
```

如果你之前手动改过，关闭前端重新启动一次最稳。

### 8.5 当前仓库 `tsc` 不是全绿

目前项目里还存在与本次前端跑通无关的历史类型问题，因此如果你直接跑全量：

```powershell
cd frontend
cmd /c npx tsc --noEmit --incremental false
```

可能会被别的旧错误挡住。

这不影响你按本指南完成前端联调，但如果你要做 CI 级别验收，需要先单独清这些历史报错。
